# E2E Testing Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build e2e tests that verify each SDK sends correct data through the ABsmartly pipeline — from SDK event emission to API-queryable metrics.

**Architecture:** A Python orchestrator (`e2e_runner.py`) manages the full lifecycle: create experiment via CLI, instruct SDK wrappers to send real events to a test ABsmartly environment, poll the API for metrics, verify counts match, tear down the experiment. Wrappers get a new `mode: "e2e"` code path that uses real SDK credentials from environment variables.

**Tech Stack:** Python 3 (orchestrator), ABsmartly CLI (`abs`), Docker Compose (wrapper containers), requests (HTTP polling)

**Spec:** `docs/specs/2026-03-20-e2e-testing-design.md`

### Task 2: E2E Runner — Experiment Lifecycle

**Files:**
- Create: `orchestrator/e2e_runner.py`

- [ ] **Step 1: Create e2e_runner.py with CLI integration**

The runner needs to:
1. Shell out to `abs` CLI to create, start, stop, and archive experiments
2. Parse experiment ID from CLI output
3. Handle errors gracefully (cleanup in finally block)

Read the CLI help to determine exact flags:
```bash
cd ~/git_tree/absmartly-cli-ts-api-client && node dist/index.js experiments create --help
cd ~/git_tree/absmartly-cli-ts-api-client && node dist/index.js experiments start --help
```

Implement:
- `generate_run_id()` — returns 8-char hex UUID prefix
- `E2ERunner.__init__(sdks, config)` — stores SDK URLs and config
- `E2ERunner.create_experiment()` — runs `abs experiments create` with correct flags, parses ID
- `E2ERunner.start_experiment()` — runs `abs experiments start {id}`
- `E2ERunner.cleanup_experiment()` — runs `abs experiments stop` then `archive`, catches all errors

The CLI requires:
- `--name` — experiment name
- `--application-id` — numeric application ID (look up from name using `abs applications list`)
- `--unit-type` — numeric unit type ID (look up from name using `abs unit-types list`)
- `--variants` — comma-separated variant names
- `--env` — environment name
- `--profile` — CLI profile

- [ ] **Step 2: Test experiment lifecycle manually**

Run the create/start/stop/archive sequence manually to verify:
```bash

abs experiments start {id} --profile e2e
abs experiments stop {id} --profile e2e

```

### Task 3: E2E Runner — SDK Scenario Execution

**Files:**
- Modify: `orchestrator/e2e_runner.py`

- [ ] **Step 1: Implement run_sdk_scenarios**

For each SDK wrapper (in batches of 5 to avoid overwhelming the API):
1. `POST /context` with `{"mode": "e2e", "units": {"session_id": "e2e-{run_id}-{sdk}-{i}"}, "attributes": {"sdk_name": "{sdk}"}}`
2. `POST /context/{id}/treatment` with `{"experimentName": "e2e-{run_id}"}`
3. For units 0..49: `POST /context/{id}/track` with `{"goalName": "purchase", "properties": {"amount": 10}}`

Track successes and failures per SDK. Retry publish up to 3 times on transient errors (5xx, connection error).

### Task 8: JavaScript Wrapper E2E Mode (Reference Implementation)

**Files:**
- Modify: `javascript-wrapper/server.js`

This is the reference implementation. All other wrappers follow this pattern.

- [ ] **Step 1: Add e2e mode to createContext handler**

In the `POST /context` handler, when `req.body.mode === "e2e"`:
1. Read `ABSMARTLY_E2E_ENDPOINT`, `ABSMARTLY_E2E_API_KEY`, `ABSMARTLY_E2E_APPLICATION`, `ABSMARTLY_E2E_ENVIRONMENT` from `process.env`
2. If any are missing, return 501 `{"error": "e2e mode not configured"}`
3. Create a real SDK instance: `new absmartly.SDK({endpoint, apiKey, application, environment})`
4. Create context with `sdk.createContext(config)` using the provided units
5. Set attributes from `req.body.attributes`
6. Wait until ready
7. Return context as usual

Key difference from normal mode: uses `DefaultContextPublisher` (real HTTP) instead of `CustomPublisher` (no-op).

- [ ] **Step 2: Test manually**

```bash

curl -X POST http://localhost:3001/context -H 'Content-Type: application/json' \
  -d '{"mode":"e2e","units":{"session_id":"test-1"},"attributes":{"sdk_name":"javascript"}}'
```

### Task 9: Remaining Wrapper E2E Modes

**Files:**
- Modify: All other 19 wrapper servers

- [ ] **Step 1: Add e2e mode to each wrapper**

For each wrapper, add the same pattern as the JavaScript reference:
- Read e2e env vars
- If missing, return 501
- Create real SDK instance with real publisher
- Set attributes
- Return context

Group by language similarity:
1. **Node.js wrappers** (angular, react, vue2, vue3): same pattern as JavaScript
2. **Python wrapper**: `ABsmartly(ABsmartlyConfig(endpoint=..., api_key=..., ...))`
3. **Ruby/Liquid wrappers**: `ABsmartly.new(config)`
4. **Java wrapper**: `ABSmartly.create(ABSmartlyConfig.create().setClient(...))`
5. **Kotlin wrapper**: `ABSmartly.create(ABSmartlyConfig.create().setClient(...))`
6. **Go wrapper**: `sdk.Create(sdk.ABSmartlyConfig{...})`
7. **Swift wrapper**: `ABSmartlySDK(config: ABSmartlyConfig(...))`
8. **Dart/Flutter wrappers**: `ABSmartly.create(ABSmartlyConfig.create()...)`
9. **Dotnet wrapper**: `new ABsmartly(new ABsmartlyConfig {...})`
10. **PHP wrapper**: `new ABsmartly(new Config(...))`
11. **Scala wrapper**: `new SDK(SDKConfig(...))`
12. **Elixir wrapper**: `ABSmartly.start_link(config)`
13. **Rust wrapper**: `ABsmartly::new(config)`
14. **C++ wrapper**: `SDK(SDKConfig{...})`

Each wrapper reads env vars using its language's standard mechanism:
- Node.js: `process.env.ABSMARTLY_E2E_ENDPOINT`
- Python: `os.environ.get("ABSMARTLY_E2E_ENDPOINT")`
- Ruby: `ENV["ABSMARTLY_E2E_ENDPOINT"]`
- Java: `System.getenv("ABSMARTLY_E2E_ENDPOINT")`
- Go: `os.Getenv("ABSMARTLY_E2E_ENDPOINT")`
- etc.

Use team-based parallel execution: spawn one agent per 4-5 wrappers.
